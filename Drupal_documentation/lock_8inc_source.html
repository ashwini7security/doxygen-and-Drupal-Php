00204   // Pause the process for short periods between calling
00205   // lock_may_be_available(). This prevents hitting the database with constant
00206   // database queries while waiting, which could lead to performance issues.
00207   // However, if the wait period is too long, there is the potential for a
00208   // large number of processes to be blocked waiting for a lock, especially
00209   // if the item being rebuilt is commonly requested. To address both of these
00210   // concerns, begin waiting for 25ms, then add 25ms to the wait period each
00211   // time until it reaches 500ms. After this point polling will continue every
00212   // 500ms until $delay is reached.
00213 
00214   // $delay is passed in seconds, but we will be using usleep(), which takes
00215   // microseconds as a parameter. Multiply it by 1 million so that all
00216   // further numbers are equivalent.

00218 
00219   // Begin sleeping at 25ms.

00223     // lock, so we sleep first to give the parallel request a chance to finish
